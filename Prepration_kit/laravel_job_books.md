Here‚Äôs a clear and concise explanation of the difference between `INNER JOIN`, `LEFT JOIN`, and `RIGHT JOIN` in MySQL:

---

### üîÅ **INNER JOIN**

* **Returns**: Only the **matching rows** from both tables.
* **Use case**: You want records where there's a match in both tables.

```sql
SELECT * 
FROM employees e
INNER JOIN departments d ON e.department_id = d.id;
```

> ‚úÖ Result: Only employees who belong to a department.

---

### ‚¨ÖÔ∏è **LEFT JOIN** (or LEFT OUTER JOIN)

* **Returns**: **All rows** from the **left table**, and matching rows from the right table. If no match, right table columns are `NULL`.
* **Use case**: You want **all employees**, even those without a department.

```sql
SELECT * 
FROM employees e
LEFT JOIN departments d ON e.department_id = d.id;
```

> ‚úÖ Result: Every employee, with or without a department.

---

### ‚û°Ô∏è **RIGHT JOIN** (or RIGHT OUTER JOIN)

* **Returns**: **All rows** from the **right table**, and matching rows from the left table. If no match, left table columns are `NULL`.
* **Use case**: You want **all departments**, even if no employee is assigned.

```sql
SELECT * 
FROM employees e
RIGHT JOIN departments d ON e.department_id = d.id;
```

> ‚úÖ Result: Every department, even if it has no employees.

---

### üìå Summary Table

| Join Type    | Matched from Both | All from Left | All from Right |
| ------------ | ----------------- | ------------- | -------------- |
| `INNER JOIN` | ‚úÖ Yes             | ‚ùå No          | ‚ùå No           |
| `LEFT JOIN`  | ‚úÖ Yes             | ‚úÖ Yes         | ‚ùå No           |
| `RIGHT JOIN` | ‚úÖ Yes             | ‚ùå No          | ‚úÖ Yes          |

### ‚úÖ **Difference Between `var`, `let`, and `const` in JavaScript**

These three keywords are used to **declare variables**, but they differ in scope, hoisting, and mutability.

---

### üîπ **1. `var`**

* **Function-scoped** (not block-scoped)
* **Hoisted** (but initialized as `undefined`)
* Can be **redeclared** and **reassigned**

```javascript
var x = 10;
if (true) {
  var x = 20;  // same variable, updated
}
console.log(x); // ‚ûú 20
```

---

### üîπ **2. `let`**

* **Block-scoped**
* **Hoisted**, but **not initialized** (access before declaration causes `ReferenceError`)
* Can be **reassigned**, but **not redeclared** in the same scope

```javascript
let y = 10;
if (true) {
  let y = 20;  // new variable (block-scoped)
  console.log(y); // ‚ûú 20
}
console.log(y); // ‚ûú 10
```

---

### üîπ **3. `const`**

* **Block-scoped**
* Must be **initialized at declaration**
* Cannot be **reassigned** or **redeclared**
* For objects and arrays, **contents can change**, but **reference can't**

```javascript
const z = 10;
// z = 20; // ‚ùå Error: Assignment to constant variable

const user = { name: "Sandeep" };
user.name = "Om";  // ‚úÖ Allowed (object property changed)
```

---

### üß† Summary Table:

| Feature        | `var`           | `let`     | `const`   |
| -------------- | --------------- | --------- | --------- |
| Scope          | Function        | Block     | Block     |
| Hoisted        | Yes (undefined) | Yes (TDZ) | Yes (TDZ) |
| Redeclarable   | ‚úÖ Yes           | ‚ùå No      | ‚ùå No      |
| Reassignable   | ‚úÖ Yes           | ‚úÖ Yes     | ‚ùå No      |
| Initialization | Optional        | Optional  | Required  |

### ‚úÖ **What is Hoisting in JavaScript?**

**Hoisting** is JavaScript‚Äôs default behavior of **moving declarations to the top** of their scope (before code execution).

But there's a catch:

---

### üîπ **1. Variable Hoisting**

* Variables declared with `**var**` are **hoisted** ‚Äî but **only the declaration**, **not the assignment**.
* Variables declared with `**let**` and `**const**` are **hoisted** too, but placed in the **Temporal Dead Zone (TDZ)** ‚Äî accessing them before declaration causes an error.

```javascript
console.log(x); // undefined (not ReferenceError)
var x = 5;
```

```javascript
console.log(y); // ‚ùå ReferenceError
let y = 10;
```

---

### üîπ **2. Function Hoisting**

* **Function declarations** are hoisted **with their body**, so you can call them before they're defined:

```javascript
greet(); // ‚úÖ Works
function greet() {
  console.log('Hello!');
}
```

* **Function expressions** (especially with `const` or `let`) are **not fully hoisted**:

```javascript
sayHi(); // ‚ùå TypeError: sayHi is not a function
const sayHi = function() {
  console.log('Hi!');
};
```

---

### üß† Summary Table:

| Type                 | Hoisted? | Initialized? | Access Before Declaration |
| -------------------- | -------- | ------------ | ------------------------- |
| `var`                | ‚úÖ Yes    | ‚ùå No         | ‚úÖ undefined               |
| `let` / `const`      | ‚úÖ Yes    | ‚ùå No         | ‚ùå ReferenceError (TDZ)    |
| Function Declaration | ‚úÖ Yes    | ‚úÖ Yes        | ‚úÖ Allowed                 |
| Function Expression  | ‚ùå No     | ‚ùå No         | ‚ùå Error                   |

### ‚úÖ **How PHP Handles Sessions Using `session_start()` and `$_SESSION`**

Sessions in PHP allow you to **store user-specific data across multiple pages**, like login state, cart items, etc.

---

### üîπ **1. `session_start()`**

* Initializes a **new session** or resumes an **existing one**
* Must be called **at the top** of the script (before any output)

```php
<?php
session_start();
?>
```

> ‚úÖ Creates a session ID and stores it in a cookie (`PHPSESSID`)

---

### üîπ **2. Storing Data in `$_SESSION`**

```php
<?php
session_start();
$_SESSION['username'] = 'Sandeep';
$_SESSION['role'] = 'admin';
?>
```

> ‚úÖ Values are stored on the server and available across all pages using the same session.

---

### üîπ **3. Accessing Session Data**

```php
<?php
session_start();
echo $_SESSION['username']; // ‚ûú Sandeep
?>
```

---

### üîπ **4. Removing Session Data**

* **Unset a variable**:

```php
unset($_SESSION['username']);
```

* **Destroy the entire session**:

```php
session_destroy();
```

> ‚ö†Ô∏è After `session_destroy()`, the session is deleted but variables may still exist until the script ends.

---

### üß† Common Use Case: **Login System**

```php
// login.php
session_start();
$_SESSION['logged_in'] = true;
$_SESSION['user_id'] = 101;
```

```php
// dashboard.php
session_start();
if ($_SESSION['logged_in'] !== true) {
  header('Location: login.php');
  exit;
}
```

---

### üìå Summary

| Function            | Purpose                               |
| ------------------- | ------------------------------------- |
| `session_start()`   | Starts or resumes a session           |
| `$_SESSION`         | Stores and accesses session variables |
| `unset()`           | Removes a specific session variable   |
| `session_destroy()` | Deletes the session completely        |
### ‚úÖ **Connecting to a MySQL Database Using `mysqli` in PHP and Handling Errors**

You can use the **`mysqli`** extension in PHP to connect to a MySQL database.

---

### üîπ **1. Basic Connection Code**

```php
<?php
$host = "localhost";
$user = "root";
$password = "";
$database = "my_database";

// Create connection
$conn = new mysqli($host, $user, $password, $database);

// Check connection
if ($conn->connect_error) {
    die("‚ùå Connection failed: " . $conn->connect_error);
}

echo "‚úÖ Connected successfully";
?>
```

---

### üîç **Explanation:**

* `new mysqli(...)`: Opens a new connection.
* `$conn->connect_error`: Returns error message if the connection fails.
* `die()`: Stops the script if connection fails.

---

### üîπ **2. Safer Version with Error Reporting**

```php
<?php
$mysqli = new mysqli("localhost", "root", "", "my_database");

if ($mysqli->connect_errno) {
    echo "‚ùå Failed to connect to MySQL: (" . $mysqli->connect_errno . ") " . $mysqli->connect_error;
    exit();
}
echo "‚úÖ Connection OK!";
?>
```

---

### üß† Best Practice:

* Keep credentials in a separate `config.php` file.
* Always check for errors before running queries.
* Close the connection when done:

```php
$conn->close();
```

### ‚úÖ **Associative Arrays in PHP & Looping with `foreach`**

---

### üîπ **What is an Associative Array?**

An **associative array** in PHP uses **named keys** instead of numeric indexes.

```php
$user = [
    "name" => "Sandeep",
    "email" => "sandeep@example.com",
    "role" => "Developer"
];
```

> ‚úÖ Keys like `"name"`, `"email"`, and `"role"` are strings, not numbers.

---

### üîπ **Looping Through Associative Arrays Using `foreach`**

#### üî∏ Loop Through **Key + Value**:

```php
foreach ($user as $key => $value) {
    echo "$key: $value <br>";
}
```

> ‚úÖ Output:

```
name: Sandeep
email: sandeep@example.com
role: Developer
```

---

#### üî∏ Loop Through **Values Only**:

```php
foreach ($user as $value) {
    echo "$value <br>";
}
```

> ‚úÖ Output:

```
Sandeep
sandeep@example.com
Developer
```

---

### üß† Tip:

You can nest associative arrays and loop through them too:

```php
$users = [
    "u1" => ["name" => "Sandeep", "role" => "Dev"],
    "u2" => ["name" => "Om", "role" => "Designer"]
];

foreach ($users as $id => $info) {
    echo "$id: " . $info["name"] . " - " . $info["role"] . "<br>";
}
```
